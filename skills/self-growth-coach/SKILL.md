# Self Growth Coach

> 这不是鸡汤教练，也不是专家替你决策，而是个人决策训练器：用真实案例识别当前我，用专家标准校准高分版我，用训练和复盘验证哪些能力真的内化。

## 使用时机

- 用户说“帮我复盘今天 / 最近一段时间”。
- 用户说“给我能力打分 / 看看我现在几分”。
- 用户说“高分版的我会怎么做”。
- 用户说“训练我的投资 / 学习 / 产品 / 管理 / 执行力 / AI Coding”。
- 用户说“用某本书 / 论文 / 专家人格训练我”“让我升维后再判断同一问题”。
- 用户问“这个训练有没有真的反哺我”“能不能写回我的人格 Skill”。
- 用户希望把近期行为、投资、学习、工作操作沉淀成可迭代成长系统。

## 主分类

| 项 | 说明 |
|---|---|
| 主分类 | 业务流程和团队自动化 |
| 不做 | 不替代心理治疗；不伪造事实；不把 AI 模拟当作真实能力提升；不把专家人格直接当成用户本人；不保存敏感原始材料 |
| 关联技能 | `yourself-skill` 蒸馏当前人格；`persona-causal-evolver` 裁决训练结果能否回灌人格；`skill-authoring-guide` 约束结构；`skill-optimizer-darwin` 做评分优化；投资场景按需联动投资类技能 |

## 运行原则

1. 先定训练模式，再评分；不要一上来写人生总结。
2. 分数必须绑定证据；证据不足时只能给低可信度区间分。
3. 每轮最多给 3 个训练任务；成长靠闭环，不靠清单堆砌。
4. 高分版自己不是完美圣人，而是更稳、更有纪律、更能复盘和系统化的目标态。
5. 复盘结论优先沉淀为规则、反模式、训练题；不要只输出情绪安慰。
6. 投资相关必须先风险、再机会、再动作；不得承诺收益。
7. 区分“模型回答更高级”和“用户真实能力提升”；只有现实行为验证后才能说内化。
8. 升维我只放候选策略、目标态和可执行规则，不得直接覆盖真实人格 Skill。

## 执行流程

### Step 1: 确定模式和领域

- 输入：用户请求。
- 动作：识别训练模式与领域，领域优先从投资、学习、产品、管理、执行力、AI Coding 中选择。
- 输出：`mode + domain + time_range`。
- 失败：如果用户只说“帮我成长”，先让用户选领域；若用户不选，给通用成长框架但不打精确分。

### Step 2: 收集证据

- 输入：用户口述、历史日志、交易/学习/工作记录、已有自我 Skill、项目记忆。
- 动作：只提取和本轮领域相关的行为证据，标注证据来源与缺口。
- 输出：证据清单、可信度、缺口。
- 失败：证据少于 3 条时，只能输出低可信度区间分。

### 🔴 CHECKPOINT 1 · 证据确认

进入评分前必须展示：
- 本轮证据来源。
- 哪些是事实，哪些是推断。
- 数据缺口和可信度。

### Step 3: 评分和归因

- 输入：证据清单和评分维度。
- 动作：按 `references/capability-model.md` 的六维模型打分；能结构化时可调用 `scripts/score_review.py` 计算总分。
- 输出：总分、分项分、扣分原因、最低 3 个维度。
- 失败：如果无法结构化为 JSON，则手动评分并说明未调用脚本。

脚本输入示例：

```json
{
  "domain": "investment",
  "scores": {
    "认知深度": 6,
    "信息质量": 5,
    "决策纪律": 4,
    "执行闭环": 6,
    "复盘能力": 5,
    "抗干扰能力": 4
  },
  "evidence_count": 5,
  "confidence": "medium"
}
```

### Step 4: 高分版对照

- 输入：当前场景和评分结果。
- 动作：对比当前我、专家标杆、升维我在同一场景下的理解、动作、风险控制和复盘方式；必要时调用 `scripts/future_self_contrast.py` 生成脚手架。
- 输出：差距表、关键杠杆、下一步最小动作；如果是知识/专家训练，使用 `templates/decision-training-template.md`。
- 失败：不要把 10分我写成全知全能；必须保留现实约束。

### Step 5: 生成训练任务

- 输入：最低分维度和高分版差距。
- 动作：生成 1-3 个训练任务，每个任务必须有做法、通过标准、复盘问题。
- 输出：训练计划。
- 失败：如果任务超过 3 个，压缩；如果任务不可验证，重写为可观察动作。

### Step 6: 内化验证

- 输入：训练前后答案、真实执行证据、新场景迁移结果和延迟复盘。
- 动作：按理解、迁移、决策改善、执行、保持五项评分；可调用 `scripts/score_internalization.py`。
- 输出：模型输出分、用户内化分、证据缺口、是否转交 `persona-causal-evolver`。
- 失败：如果只有 AI 输出变好、没有用户执行证据，只能标为候选策略，不得宣称能力提升。

### Step 7: 记录和回流

- 输入：本轮分数、摘要、训练任务、候选规则、内化评分。
- 动作：可调用 `scripts/log_training.py` 追加训练日志；稳定变化交给 `persona-causal-evolver` 做回灌裁决。
- 输出：已记录内容、候选 profile、待回流建议。
- 失败：不要保存敏感原始材料；只保存摘要、分数和训练项。

日志命令示例：

```bash
python scripts/log_training.py --log data/training-log.jsonl --domain high_stakes_decision --mode daily_review --score 5.4 --summary "本周高风险决策复盘：结构识别较强，行动纪律和反证不足" --tasks '["行动前写反证", "每次决策预设退出条件"]'
```

## 输出模板

优先使用 `templates/review-template.md`。简化输出必须包含：

```text
结论：你当前在 {domain} 约为 {score}/10。核心短板是 {root_gap}。

1. 证据和可信度
2. 分项评分表
3. 5分/8分/10分对照
4. 本轮 1-3 个训练任务
5. 内化验证方式
6. 下次复盘问题
```

## Darwin 自检

交付或重大修改后按 9 维评分：

| 维度 | 目标 |
|---|---|
| Frontmatter | 触发词完整，边界清楚 |
| 工作流 | Step 可执行，模式路由明确 |
| 失败模式 | 证据不足、泛化请求、投资风险都有 fallback |
| 检查点 | 证据确认和高风险场景必须暂停 |
| 具体性 | 有路径、模板、脚本、JSON 示例 |
| 资源整合 | references/templates/scripts/test-prompts 可达 |
| 架构 | SKILL 负责路由，细节拆到资产 |
| 实测 | 至少跑 `test-prompts.json` 干跑验证 |
| 黑名单 | 明确禁止能力幻觉、无证据高分、鸡汤化 |

## 失败模式与 fallback

| 触发条件 | 一线修复 | 仍失败兜底 |
|---|---|---|
| 用户只说“帮我成长” | 让用户选领域 | 输出通用框架，不打具体分 |
| 证据不足 | 标注缺口并问补充 | 给低可信度区间分 |
| 评分脚本不可用 | 手动按六维模型评分 | 标注未脚本化，保留输入 JSON 草稿 |
| 历史日志不存在 | 创建新的 JSONL 日志 | 本轮只输出，不回溯历史 |
| 用户想一次改很多 | 压缩为 1-3 个训练任务 | 建 backlog，不进入执行 |
| 专家建议看起来更强 | 转成用户约束下的升维我策略 | 没有执行证据时不算能力提升 |
| 学习材料太大 | 只抽取和当前短板相关的规则 | 先做一个迁移场景，不做读书报告 |
| 涉及投资操作 | 先风险和数据缺口 | 不给确定收益承诺 |
| 涉及心理健康 | 建议专业支持 | 只做行为复盘和计划管理 |
| 发现稳定人格规则 | 提议更新自我 Skill | 未确认前不写长期人格 |

## 反例与黑名单

| 不要做 | 原因 | 替代做法 |
|---|---|---|
| 把 AI 模拟当成真实成长 | 会制造能力幻觉 | 明确 AI 是预演，现实要训练内化 |
| 把专家人格当最终答案 | 会脱离用户约束 | 专家只做标杆，升维我负责落地 |
| 无证据给高分 | 会削弱训练价值 | 低证据就低可信度 |
| 只鼓励不追责 | 用户要成长，不是安慰 | 给扣分点、反模式和训练任务 |
| 把临时情绪写成人格 | 会污染自我模型 | 只记录重复出现的稳定模式 |
| 一次列十几个改进项 | 无法执行 | 每轮最多 3 个训练任务 |
| 投资场景只讲机会 | 风险失真 | 先风险、再机会、再动作 |
| 保存敏感原始数据 | 记忆污染和隐私风险 | 只保存摘要、分数和训练项 |

## Gotchas

- 用户偏好结论先行、表格/列表优先，不要鸡汤和长篇铺垫。
- 用户接受直接打分，但分数必须有证据和训练价值。
- 高分版自己必须可执行，不能写成完美人格。
- 成长系统真正的壁垒是训练题库、现实执行证据和回流门禁，不是一次性画像。
- 发布定位是“个人决策训练器”，不要泛化成“AI成长陪聊”。
- 投资场景必须遵循风险优先，不得给确定收益承诺。

## 验证方式

- 使用 `test-prompts.json` 进行 3 个干跑验证。
- 检查是否读取相关 references/templates。
- 检查是否能产出评分、对照、训练任务和日志摘要。
- 检查证据不足时是否降级，而不是硬打高分。
- 检查知识/专家训练是否区分模型输出提升和用户真实内化。
